# 自动化执行策略规则

## 规则说明

此规则定义在执行开发任务时的自动化策略，根据操作风险等级选择相应的执行方式。

**规则类型**: Model Decision  
**触发场景**: 当 AI 需要决定如何执行开发任务时自动应用

---

## 风险等级定义

### 🟢 低风险（自动执行）

**特征**:
- 不可逆性：低
- 影响范围：小
- 恢复难度：易
- 置信度要求：> 80%

**典型操作**:
- ✅ 读取文件内容
- ✅ 创建新文件
- ✅ 更新 spec 文档（添加实施笔记、标记任务完成）
- ✅ 运行测试
- ✅ 代码格式化
- ✅ 添加注释

**执行策略**: 
- 直接执行，无需确认
- 记录到 Git commit
- 更新 spec 实施笔记

---

### 🟡 中风险（创建快照后执行）

**特征**:
- 不可逆性：中
- 影响范围：中
- 恢复难度：中（可通过 Git 回滚）
- 置信度要求：> 60%

**典型操作**:
- ⚠️ 修改现有文件
- ⚠️ 删除单个非关键文件
- ⚠️ Git 提交和推送
- ⚠️ 重构代码
- ⚠️ 更新依赖

**执行策略**:
1. 创建 Git branch（如果还没有）
2. 执行操作
3. 验证结果
4. 如果成功：合并 branch
5. 如果失败：丢弃 branch，报告错误

**示例工作流**:
```bash
# 1. 创建分支
git checkout -b task/Task-XXX-description

# 2. 执行修改
# ... 修改文件 ...

# 3. 验证
npm test  # 或相应的测试命令

# 4. 提交
git add .
git commit -m "task: Task-XXX [描述]"

# 5. 如果验证通过，稍后合并；否则丢弃分支
```

---

### 🟠 高风险（询问用户）

**特征**:
- 不可逆性：高
- 影响范围：大
- 恢复难度：难
- 置信度要求：任意

**典型操作**:
- ❓ 删除多个文件
- ❓ 修改配置文件（package.json, Cargo.toml 等）
- ❓ 数据库 schema 变更
- ❓ API 接口修改
- ❓ 架构调整

**执行策略**:
1. 列出将要执行的操作
2. 说明潜在风险
3. 提供选项：
   - A: 继续执行
   - B: 修改方案
   - C: 取消操作
4. 等待用户确认
5. 根据用户选择执行

**提问模板**:
```markdown
❓ **需要您的确认**

我计划执行以下操作：

**操作列表**:
1. 删除文件: file1.txt, file2.txt
2. 修改配置: package.json (添加依赖)

**潜在风险**:
- 删除的文件可能仍被引用
- 新依赖可能与其他包冲突

**建议**:
- 先运行测试确保无破坏性更改
- 审查依赖兼容性

请选择：
- A: 继续执行（我会先运行测试）
- B: 修改方案（请告诉我如何调整）
- C: 取消操作

等待您的回复...
```

---

### 🔴 严重风险（需要明确授权）

**特征**:
- 不可逆性：极高
- 影响范围：极大
- 恢复难度：极难
- 置信度要求：必须 100% 确定

**典型操作**:
- 🚫 删除重要目录（src/, tests/ 等）
- 🚫 部署到生产环境
- 🚫 数据迁移或删除
- 🚫 修改核心架构
- 🚫 批量删除 Git commits

**执行策略**:
1. 详细说明操作的后果
2. 强调不可逆性
3. 要求用户输入明确的授权词（如 "APPROVE" 或 "CONFIRM"）
4. 创建完整的备份
5. 仅在收到明确授权后执行
6. 执行后验证并报告

**授权模板**:
```markdown
🚨 **严重风险操作警告**

您请求的操作具有极高风险且不可逆：

**操作**: [详细描述]

**后果**:
- [后果 1]
- [后果 2]
- [后果 3]

**无法恢复**: 一旦执行，将无法撤销

**如需继续**, 请输入: **APPROVE**

或者输入其他内容以取消此操作。

⚠️ 请注意：只有输入 "APPROVE" 才会执行此操作。
```

## 决策流程

```
收到任务
   ↓
评估风险等级
   ↓
┌─────────────┐
│  低风险？    │ ──── YES ──→ 自动执行
└─────┬───────┘
      │ NO
      ↓
┌─────────────┐
│  中风险？    │ ──── YES ──→ 创建 Git branch → 执行 → 验证
└─────┬───────┘
      │ NO
      ↓
┌─────────────┐
│  高风险？    │ ──── YES ──→ 询问用户 → 等待确认 → 执行
└─────┬───────┘
      │ NO
      ↓
  严重风险 ────────→ 要求明确授权 → 备份 → 执行
```

## 特殊场景处理

### 场景 1: 批量操作

**原则**: 批量操作的风险 = 单个操作风险 × 数量

**示例**:
- 删除 1 个文件：中风险
- 删除 10 个文件：高风险
- 删除 50 个文件：严重风险

**策略**:
- 对于批量操作，提升一个风险等级
- 提供详细的影响分析
- 建议分批次执行

### 场景 2: 连锁反应

**原则**: 考虑操作的间接影响

**示例**:
- 修改 API 接口 → 可能需要更新所有调用方
- 删除公共函数 → 可能破坏多个模块

**策略**:
- 使用 `#codebase` 进行全局检索
- 识别所有受影响的位置
- 在风险评估中包含间接影响

### 场景 3: 不确定性

**原则**: 不确定时，提升风险等级

**策略**:
- 如果置信度 < 60%，至少提升到高风险
- 主动询问用户澄清
- 不要假设或猜测

## 最佳实践

### 1. 优先使用 Git

- ✅ 每个任务开始前创建 branch
- ✅ 频繁提交小的变更
- ✅ 使用描述性的 commit message
- ✅ 失败时丢弃 branch，而非强制回滚

### 2. 渐进式执行

- ✅ 先执行低风险部分
- ✅ 验证后再执行中风险部分
- ✅ 最后处理高风险部分
- ✅ 每步都报告进度

### 3. 透明沟通

- ✅ 在执行前说明计划
- ✅ 执行中报告进度
- ✅ 执行后总结结果
- ✅ 遇到问题立即报告

### 4. 安全第一

- ✅ 宁可多问一次，也不要冒险
- ✅ 不确定时选择更保守的策略
- ✅ 始终保留回滚能力
- ✅ 记录所有决策和理由

## 与 Spec 的集成

### 在 Spec 中标记风险

```markdown
### Task-005: 创建阈值配置 UI 组件

**风险等级**: 🟡 中风险
**原因**: 修改现有组件，可能影响其他功能

**执行策略**:
1. 创建 branch: feature/threshold-ui
2. 实现组件
3. 运行测试
4. 如果通过，提交并准备合并
```

### 在实施笔记中记录

```markdown
### 实施笔记 - 2024-01-15 17:30

**执行策略**: 中风险 - 创建 Git branch
**Branch**: feature/threshold-ui
**验证**: 测试全部通过 (12/12)
**结果**: 成功，准备合并
```

## 示例应用场景

### 示例 1: 创建新文件（低风险）

**用户**: "创建一个 README.md 文件"

**AI 决策**:
- 风险等级: 🟢 低风险
- 策略: 自动执行
- 行动: 直接创建文件，提交 Git

### 示例 2: 重构函数（中风险）

**用户**: "重构这个函数以提高性能"

**AI 决策**:
- 风险等级: 🟡 中风险
- 策略: 创建 branch → 重构 → 运行测试 → 验证
- 行动:
  ```bash
  git checkout -b refactor/optimize-function
  # 执行重构
  npm test
  # 如果通过，稍后合并
  ```

### 示例 3: 删除废弃代码（高风险）

**用户**: "删除所有未使用的函数"

**AI 决策**:
- 风险等级: 🟠 高风险
- 策略: 询问用户
- 行动:
  ```markdown
  ❓ **需要您的确认**
  
  我检测到 15 个未使用的函数：
  - func1 (在 module_a)
  - func2 (在 module_b)
  - ...
  
  删除这些函数可能导致：
  - 如果有外部调用，会报错
  - 如果文档中有引用，会过时
  
  建议：
  1. 先运行全局搜索确认无引用
  2. 分批删除，每次删除后测试
  
  是否继续？
  - A: 继续（我会先检查引用）
  - B: 手动审查每个函数
  - C: 取消
  ```

### 示例 4: 重置数据库（严重风险）

**用户**: "清空数据库重新开始"

**AI 决策**:
- 风险等级: 🔴 严重风险
- 策略: 要求明确授权
- 行动:
  ```markdown
  🚨 **严重风险操作警告**
  
  您请求的操作将清空整个数据库：
  
  **后果**:
  - 所有数据永久丢失
  - 无法恢复（除非有备份）
  - 可能影响其他系统
  
  **如需继续**, 请输入: **APPROVE**
  
  或者输入其他内容以取消。
  ```

## 总结

此规则确保：

✅ **风险评估标准化** - 清晰的四级风险分类  
✅ **执行策略明确** - 每种风险对应明确的策略  
✅ **用户控制适当** - 高风险操作需要确认  
✅ **安全第一** - 宁可保守，不要冒险  
✅ **Git 集成** - 充分利用版本控制  

**记住**: 当不确定时，选择更保守的策略，并与用户沟通。
