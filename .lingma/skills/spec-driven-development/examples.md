## 完整示例：文件夹阈值告警功能

### Step 1: 需求澄清

AI 会问：

```
📋 需求澄清

为了创建准确的 spec，请回答以下问题：

1. 阈值配置方式？
   ○ 固定值（如 1GB）
   ○ 用户可配置
   ○ 不同文件夹不同阈值

2. 告警方式？
   ○ UI 通知（Toast/Alert）
   ○ 系统通知
   ○ 邮件通知
   ○ 全部

3. 检查频率？
   ○ 实时监控（文件变化时）
   ○ 定时检查（每小时/每天）
   ○ 手动触发

4. 优先级？
   ○ P0 - 紧急
   ○ P1 - 高
   ○ P2 - 中
   ○ P3 - 低
```

### Step 2: Spec 草案

AI 创建 `.lingma/specs/current-spec.md`：

```markdown
# 文件夹大小阈值告警功能 Spec

## 元数据
- **创建日期**: 2024-01-15
- **状态**: draft
- **优先级**: P1
- **负责人**: 开发团队

## 背景与目标
### 问题陈述
用户无法及时了解文件夹大小是否超出预期，可能导致磁盘空间不足。

### 业务价值
- 预防磁盘空间耗尽导致的服务中断
- 提高用户对存储使用的可见性
- 减少运维成本

### 成功标准
- [ ] 用户可以在 3 次点击内配置阈值
- [ ] 告警延迟 < 5 秒
- [ ] 误报率 < 1%

## 需求规格
### 功能性需求

#### FR-001: 阈值配置
**描述**: 用户可以为主题监控的文件夹设置大小阈值

**验收标准**:
- [ ] AC-001: 支持设置绝对值（MB/GB）
- [ ] AC-002: 支持设置相对值（磁盘百分比）
- [ ] AC-003: 配置立即生效，无需重启

**优先级**: Must have

---

#### FR-002: 实时监测
**描述**: 系统实时监测文件夹大小变化

**验收标准**:
- [ ] AC-004: 文件创建/删除时触发检查
- [ ] AC-005: 文件大小变化超过 10% 时触发检查
- [ ] AC-006: 检查耗时 < 100ms

**优先级**: Must have

---

#### FR-003: 告警通知
**描述**: 当文件夹大小超过阈值时发送告警

**验收标准**:
- [ ] AC-007: UI Toast 通知
- [ ] AC-008: 告警历史记录
- [ ] AC-009: 支持静音特定文件夹

**优先级**: Should have

## 技术方案
### 架构设计

```
前端 (React) ←→ Tauri Backend (Rust) ←→ 文件系统
     ↓                    ↓
  配置 UI           阈值检查引擎
     ↓                    ↓
  告警显示           事件聚合器
```

### 数据模型

```sql
-- 在 watched_folders 表中添加字段
ALTER TABLE watched_folders 
ADD COLUMN size_threshold_bytes BIGINT,
ADD COLUMN threshold_type VARCHAR(20) DEFAULT 'absolute'; -- 'absolute' or 'percentage'
```

### API 设计

```typescript
// 更新文件夹配置
PUT /api/folders/:id/threshold
interface UpdateThresholdRequest {
  size_threshold_bytes?: number;
  threshold_type?: 'absolute' | 'percentage';
}

// 获取告警历史
GET /api/folders/:id/alerts?limit=50
interface AlertRecord {
  id: string;
  folder_id: string;
  triggered_at: Date;
  current_size: number;
  threshold: number;
  acknowledged: boolean;
}
```

- 问题: SQLite 不支持直接 ALTER TABLE ADD 多个列

**测试结果**:
- 迁移脚本测试通过
- 向后兼容验证通过

## 常见问题处理

### Q1: Spec 太复杂怎么办？

**A**: 拆分成多个小 spec：

```
原始 spec: 完整的用户管理系统
↓ 拆分
spec-1: 用户注册和登录
spec-2: 用户资料管理
spec-3: 权限和角色管理
```

### Q2: 实现过程中发现技术不可行？

**A**: 

```markdown
### 实施笔记 - 2024-01-15 17:00

**遇到的问题**:
- 问题: 原计划的实时监控系统资源占用过高
- 影响: 无法满足 NFR-001 性能要求

**解决方案**:
- 方案 1: 改用轮询机制（5 秒间隔）
- 方案 2: 使用操作系统的文件系统通知 API

**决策**: 采用方案 2，需要修改 Task-003

**影响评估**:
- 额外工作量: +2h
- 不影响其他任务
- 验收标准仍然可满足

**行动**: 更新 spec 中的技术方案部分
```

### Q3: 多个 spec 并行怎么办？

**A**: 使用优先级管理：

```bash
# 查看当前活跃 spec
ls -la .lingma/specs/current-spec.md

# 如果有多个项目，可以：
mv current-spec.md spec-history/paused-$(date +%Y%m%d)-feature-name.md

# 切换到新 spec
# (AI 会创建新的 current-spec.md)
```

## 最佳实践总结

### ✅ Do's

1. **保持 spec 简洁**: 关注 WHAT 和 WHY，让 AI 决定 HOW
2. **及时更新**: 每次重大变更后更新 spec
3. **详细记录**: 实施笔记要包含决策理由
4. **定期回顾**: 每周检查 spec 进度和质量
5. **善用模板**: 不要从头创建，使用提供的模板

### ❌ Don'ts

1. **不要过度设计**: spec 不是技术设计文档
2. **不要忽略验收标准**: 这是质量的保证
3. **不要跳过 spec**: 即使是很小的功能
4. **不要害怕变更**: spec 是活的文档
5. **不要独享 spec**: 与团队分享和评审

## 高级技巧

对于相似的功能，可以复制历史 spec 作为起点：

```bash

### 技巧 2: 批量任务管理

对于大型项目，可以创建 parent spec 和 child specs：

```
parent-spec.md (总体架构和里程碑)
├── child-spec-001.md (模块 A)
├── child-spec-002.md (模块 B)
└── child-spec-003.md (模块 C)

## 总结

Spec-Driven Development 的核心价值：

1. **清晰的意图**: 在写代码之前明确要做什么
2. **可追溯性**: 每个代码变更都能追溯到需求
3. **自主开发**: AI 基于 spec 独立工作，减少交互
4. **质量保证**: 验收标准确保交付质量
5. **知识沉淀**: spec 历史成为项目知识库

开始使用吧！🚀
