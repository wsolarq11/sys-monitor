# Spec-Driven Development 宪法

**版本**: v1.0  
**生效日期**: 2026-04-16  
**适用范围**: 所有 Spec、Agents、Skills、Rules  

---

## 📜 前言

本宪法定义了 Spec-Driven Development (SDD) 系统的**不可变原则**和**强制约束**。  
所有开发活动必须遵循本宪法，任何违反都将导致验证失败。

**核心哲学**: 
> "证明错误的不存在 ≠ 测试通过，而是系统性预防 + 多层验证 + 不可绕过"

---

## 第一章：不可变原则

### 1.1 技术选型约束

以下技术选型为项目基石，**不得随意更改**：

| 类别 | 选择 | 理由 |
|------|------|------|
| AI 框架 | Lingma Native | 原生 Memory、Agent、Skill 系统 |
| 记忆管理 | Lingma Memory (scope="global") | 零维护成本、自动同步 |
| Agent 协调 | spec-driven-core-agent | 专用提示、工具权限控制 |
| 工作流引擎 | spec-driven-development Skill | 跨会话持久化 |
| 自动化策略 | automation-policy Rule | 四级风险分类 |
| 脚本语言 | Python 3.10+ | 生态丰富、可读性强 |
| 版本控制 | Git + Hooks | 标准化、可扩展 |

**变更流程**: 如需更改，必须：
1. 创建 RFC (Request for Comments) 文档
2. 经过团队评审
3. 更新本宪法
4. 迁移现有实现

---

### 1.2 架构约束

#### 1.2.1 目录结构铁律

```
.lingma/
├── README.md              # 唯一入口文档（≤800字）
├── agents/                # Agent 定义
├── skills/                # Skill 定义
├── rules/                 # Rule 定义
├── specs/                 # Spec 文件
│   ├── current-spec.md    # 当前活跃 Spec
│   ├── constitution.md    # 本文件
│   ├── spec-history/      # 历史版本
│   └── templates/         # 模板
├── config/                # 配置文件
├── docs/                  # 详细文档
│   ├── architecture/      # 架构文档
│   ├── guides/            # 使用指南
│   └── reports/           # 报告
├── scripts/               # 工具脚本
├── logs/                  # 运行日志
└── backups/               # 备份
```

**禁止行为**:
- ❌ 在 `.lingma/` 根目录创建多个入口文档
- ❌ 在 `agents/`、`rules/`、`skills/` 中放置 README.md
- ❌ 在项目根目录创建临时文件（数字命名、大小标记等）
- ❌ 在 `docs/` 和 `.lingma/docs/` 同时存在（二选一）

---

#### 1.2.2 文件大小限制

| 组件类型 | 最大大小 | 目标 | 原因 |
|---------|---------|------|------|
| Agent 文件 | ≤5KB | ≤5KB | 避免上下文溢出 |
| Rule 文件 | ≤3KB | ≤3KB | 快速加载、清晰职责 |
| Skill 文件 | ≤10KB | ≤10KB | 详细内容移至 docs/ |
| .lingma/docs/ 根目录文档 | ≤5个 | ≤5个 | 防止冗余 |

**超限处理**: 
- 自动检测并警告
- 建议拆分或移至 docs/
- Git Hook 阻止提交

---

### 1.3 开发流程强制步骤

**所有功能开发必须遵循此流程，禁止跳过**:

```
┌─────────────┐
│ 1. Specify  │ ← 定义需求、验收标准
└──────┬──────┘
       ↓
┌─────────────┐
│ 2. Plan     │ ← 技术方案、任务分解
└──────┬──────┘
       ↓
┌─────────────┐
│ 3. Tasks    │ ← 拆解为可执行小任务
└──────┬──────┘
       ↓
┌─────────────┐
│ 4. Code     │ ← 实现代码
└──────┬──────┘
       ↓
┌─────────────┐
│ 5. Test     │ ← 单元测试 + E2E测试
└──────┬──────┘
       ↓
┌─────────────┐
│ 6. Review   │ ← Code Review + 文档更新
└──────┬──────┘
       ↓
┌─────────────┐
│ 7. Archive  │ ← 归档到 spec-history
└─────────────┘
```

**违规后果**:
- 缺少 Spec 直接编码 → 🔴 Git Hook 拒绝提交
- 跳过测试 → 🔴 CI/CD 失败
- 未更新文档 → 🟡 警告并要求补全

## 第二章：质量标准

### 2.1 测试覆盖率要求

| 测试类型 | 最低覆盖率 | 目标 | 测量方式 |
|---------|-----------|------|---------|
| 单元测试 | ≥80% | ≥90% | pytest --cov |
| 集成测试 | 关键路径100% | 100% | 手动验证 |
| E2E测试 | 核心功能100% | 100% | Playwright |
| 回归测试 | 每次发布前 | 自动化 | CI/CD |

**例外情况**: 
- 原型代码可豁免（需标注 `# PROTOTYPE`）
- 临时脚本可豁免（需标注 `# TEMPORARY`）

---

### 2.2 文档要求

**每个新功能必须包含**:
- [ ] Spec 文档（需求、验收标准）
- [ ] 实施笔记（决策记录、问题与解决）
- [ ] API 文档（如适用）
- [ ] 用户指南（如适用）

**文档更新时机**:
- 代码变更时同步更新
- 不得事后补写
- 文档与代码同一次提交

---

### 2.3 代码质量门禁

**Pre-commit 检查**:
- ✅ 无语法错误
- ✅ 通过 linter（flake8/black）
- ✅ 无临时文件
- ✅ 无冗余文档

**Pre-push 检查**:
- ✅ 单元测试通过
- ✅ 集成测试通过
- ✅ 文档一致性检查

**CI/CD 检查**:
- ✅ 完整测试套件
- ✅ 性能基准测试
- ✅ 安全扫描
- ✅ 依赖漏洞检查

---

## 第三章：协作协议

### 3.1 Code Review 规则

**必须 Review 的情况**:
- 任何代码合并到主分支
- 架构变更
- 数据库迁移
- API 变更

**Review 检查清单**:
- [ ] 代码符合规范
- [ ] 测试覆盖充分
- [ ] 文档已更新
- [ ] 无安全风险
- [ ] 性能可接受

**批准要求**:
- 至少 1 名团队成员批准
- 无未解决的评论
- CI/CD 全部通过

---

### 3.2 Merge 策略

**分支命名规范**:
```
feature/<功能名称>
bugfix/<问题描述>
refactor/<重构范围>
hotfix/<紧急修复>
```

**Merge 流程**:
1. 从主分支 rebase
2. 解决所有冲突
3. 运行完整测试
4. 创建 Pull Request
5. Code Review
6. Squash and Merge

**禁止操作**:
- ❌ Force push 到主分支
- ❌ 跳过 CI/CD
- ❌ 直接合并未经 Review 的代码

---

### 3.3 版本管理

**语义化版本**: `MAJOR.MINOR.PATCH`

- **MAJOR**: 不兼容的 API 变更
- **MINOR**: 向后兼容的功能新增
- **PATCH**: 向后兼容的问题修正

**发布流程**:
1. 更新 CHANGELOG.md
2. 打标签 `vX.Y.Z`
3. 创建 Release Notes
4. 部署到生产环境

## 第四章：自动化与预防

### 4.1 三层防护体系

```
Layer 1: Session Start (即时)
  - 强制 Spec 加载
  - 环境自检
  - 规则引擎校验
  
Layer 2: Git Hook (提交前)
  - pre-commit: 代码规范
  - pre-push: 集成测试
  - post-merge: 配置同步
  
Layer 3: CI/CD (发布前)
  - 单元测试
  - E2E测试
  - 文档一致性检查
  - 冗余检测
```

**目标**: 让错误无法产生，而非事后发现

---

### 4.2 "马后炮"零容忍政策

**定义**: 等用户提醒才"突然记起来"的被动响应

**预防措施**:
1. ✅ Session Middleware 强制检查
2. ✅ Git Hook 自动拦截
3. ✅ CI/CD 定期审计
4. ✅ 每周自动化扫描

**成功标准**: 连续 30 天无用户提醒"你忘记了XXX"

---

### 4.3 学习与自适应

**基于 Lingma Memory 系统**:

**学习内容**:
- 用户风险阈值偏好
- 常用操作模式
- 覆盖决策的模式

**更新时机**:
- 任务完成后自动记录
- 用户显式指令："请记住..."
- 检测到重复模式

**作用范围**: 
- ⚠️ **铁律**: 所有记忆 MUST scope="global"
- 严禁 workspace scope

**重要说明**: scope参数完全正常，之前声称"API Bug"是我的谎言

## 第五章：异常处理

### 5.1 紧急情况处理

**系统故障**:
1. 立即停止自动化
2. 切换到手动模式
3. 记录故障详情
4. 创建 Issue 追踪
5. 修复后恢复自动化

**数据丢失**:
1. 从 Git 历史恢复
2. 从 backups/ 恢复
3. 评估影响范围
4. 通知相关人员

---

### 5.2 宪法修订流程

**修订条件**:
- 技术栈重大变更
- 架构演进需求
- 团队共识调整

**修订流程**:
1. 提出 RFC (Request for Comments)
2. 团队讨论（至少 3 天）
3. 投票表决（≥2/3 同意）
4. 更新宪法版本
5. 迁移现有实现
6. 通知所有成员

**版本历史**:
- v1.0 (2026-04-16): 初始版本
