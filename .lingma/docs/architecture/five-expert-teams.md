## 👥 五大专家团职责

### 1. Code Review Agent - 代码质量审查

**职责**:
- 检查代码结构、错误处理、潜在bug
- 识别性能优化点
- 提供具体修复建议
- 安全评分（0-100）

**输出示例**:
```
问题1 [High]: 临时文件检测使用宽泛Glob模式
  位置: session-middleware.py L241
  建议: 使用正则匹配替代glob

问题2 [Medium]: 异常处理过于宽泛
  位置: session-middleware.py L155
  建议: 区分不同类型的Exception
```

---

### 2. Documentation Agent - 文档完整性检查

**职责**:
- 检测缺失/冗余/不一致的文档
- 验证文档大小合规性（Rule≤3KB, Skill≤10KB）
- 识别需要拆分的大型文档
- 检查断裂引用

**输出示例**:
```
缺失链接: 112个
超大文件: 31个（>3KB的Rule文件）
重复内容: 5组
```

---

### 3. Test Runner Agent - 端到端测试执行

**职责**:
- 运行所有自动化测试（单元/集成/E2E）
- 验证核心功能通过
- 生成详细测试报告
- 诊断失败根因

**测试覆盖**:
- 单元测试: pytest（≥80%覆盖率）
- 集成测试: 关键路径100%
- E2E测试: Playwright（核心功能100%）

**输出示例**:
```
测试总数: 28项
通过: 25项 (89.3%)
失败: 2项
警告: 1项
```

---

- 跟踪Phase完成状态
- 质量门禁评估（Gate 1-5）
- 风险评估和下一步决策

**评估维度**:
- 架构完整性: 84/100
- Phase完成度: 78% (30/71任务)
- 质量门禁: Gate 1-4通过，Gate 5待实现

**输出示例**:

- 调用链清晰度从6/10提升到9/10

⚠️ 待改进:

## 🔄 执行策略

### MUST原则

**必须做到**:
- ✅ 并行启动所有5个专家团
- ✅ 持续维持专家团执行直到闭环
- ✅ 综合所有专家意见后决策
- ✅ 不清楚需求时发选择题澄清

---

### 执行流程

```
1. 并行启动5个专家团
   ↓
2. 收集所有专家报告
   ↓
3. 综合分析问题和优先级
   ↓
4. 如有不明确 → 发选择题给用户
   ↓
5. 用户确认后 → 立即执行修复
   ↓
6. 再次运行专家团验证
   ↓
7. 循环直到所有测试通过
   ↓
8. 生成最终总结报告
```

## 🎓 关键教训

1. **不要单点依赖** - 单一视角会遗漏问题
2. **并行优于串行** - 5个专家同时工作节省80%时间
3. **量化驱动决策** - 用分数和数据说话
4. **持续验证闭环** - 修复后必须再次验证
5. **选择题澄清** - 不确定时让用户选择而非猜测

## 🚫 禁止的反模式

### ❌ 反模式 1: 串行执行

```
Code Review → 完成后 → Documentation → 完成后 → Test Runner
```

**正确做法**:
```
并行启动: Code Review + Documentation + Test Runner + Supervisor + Research
```

---

### ❌ 反模式 2: 只看单一报告

```
只关注Test Runner结果，忽略Code Review和Documentation
```

**正确做法**:
```
综合5个专家团报告，识别交叉问题
```

---

### ❌ 反模式 3: 不验证修复

```
发现问题 → 修复 → 直接提交
```

**正确做法**:
```
发现问题 → 修复 → 再次运行专家团验证 → 通过后提交
```
