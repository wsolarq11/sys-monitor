## 🎯 核心使命

> **"构建一个真正自主、高效、可持续进化的 AI 辅助开发系统，让开发者专注于创造价值，而非重复劳动。"**

---

## 🌟 三大支柱

### 1. 自主性 (Autonomy)
**目标**: 最大化自动化，最小化人工干预

**关键指标**:
- ✅ 自动化覆盖率 ≥ 95%
- ✅ 平均人工介入次数 < 3次/任务
- ✅ 跨会话状态恢复率 100%

**实现方式**:
- Supervisor-Worker 架构自动编排
- 4级自主性渐进式信任
- 决策日志驱动持续优化

---

### 2. 可靠性 (Reliability)
**目标**: 确保系统在不同会话、不同模型中稳定生效

**关键指标**:
- ✅ Rules 加载成功率 100%
- ✅ Skills 识别准确率 ≥ 95%
- ✅ Agents 委派准确率 ≥ 90%
- ✅ 跨模型一致性 ≥ 85%

**实现方式**:
- `trigger: always_on` 确保 Rules 持久化
- 清晰的 description 确保 Skills 可识别
- 完整的 frontmatter 确保 Agents 可选择
- 自动化验证工具定期检查

---

### 3. 可持续性 (Sustainability)
**目标**: 系统能够自我维护、自我优化，避免臃肿

**关键指标**:
- ✅ 文档冗余率 < 5%
- ✅ 未使用组件占比 < 10%
- ✅ 每月清理完成率 100%
- ✅ 系统复杂度增长率 ≤ 5%/月

**实现方式**:
- Docs-as-Code + Self-Healing 文档系统
- 单一入口原则避免冗余
- 定期审查和清理机制
- 渐进式披露避免上下文溢出

---

## 📏 设计原则

### 1. 极简主义 (Minimalism)
> **"给 Agent 一张地图，而非一本百科全书"**

- README.md ≤ 800字
- 每个 Rule/Skill/Agent 聚焦单一职责
- 避免在 system prompt 中堆砌内容
- 按需加载，渐进式披露

---

### 2. 单一事实来源 (Single Source of Truth)
> **"一处修改，处处同步"**

- 根目录仅允许 README.md 作为入口
- 详细文档放在 docs/ 子目录
- 功能目录禁止放置 README
- 避免重复信息

---

### 3. 自动化优先 (Automation First)
> **"能自动的绝不手动"**

- Git Hook 自动检查
- CI/CD 自动验证
- 文档自动质量检测
- 冗余自动识别

---

### 4. 数据驱动 (Data-Driven)
> **"用数据说话，而非直觉"**

- 决策日志记录所有重要决策
- Metrics 监控系统健康度
- 定期分析优化方向
- 基于证据做决策

---

## 🔄 定期回顾机制

### 每周检查（5分钟）
```bash
# 运行自动化验证
python scripts/verify_system_effectiveness.py

# 检查是否有新增冗余
git status --short .lingma/
```

**检查项**:
- [ ] Rules 是否正常加载
- [ ] Skills 描述是否清晰
- [ ] Agents frontmatter 是否完整
- [ ] 是否有新的冗余文档

---

### 每月审查（30分钟）
```bash
# 分析报告冗余
python scripts/analyze_reports.py

# 检查未使用组件
# 手动审查 .lingma/ 目录下各组件的使用频率
```

**审查项**:
- [ ] 归档过时报告
- [ ] 删除未使用的 Rules/Skills
- [ ] 更新使命宣言（如需要）
- [ ] 调整设计原则（如需要）

---

### 每季度重构（2小时）
**重构项**:
- [ ] 简化系统架构
- [ ] 合并相似组件
- [ ] 优化工作流程
- [ ] 更新最佳实践

**目标**: 系统复杂度不增长，甚至降低

## 🚫 绝对禁止项

### 1. 禁止违反单一入口原则
❌ 在根目录创建多个入口文档  
❌ 在功能目录放置 README  
❌ 复制粘贴相同内容到多处  

---

### 2. 禁止过度工程化
❌ 为了"完美"而创建不必要的组件  
❌ 添加很少使用的"未来可能用到"的功能  
❌ 追求技术炫技而非实用价值  

---

### 3. 禁止忽视验证
❌ 创建 Rules/Skills/Agents 后不验证  
❌ 假设系统"应该"生效而不实际测试  
❌ 忽略自动化验证脚本的警告

## 💭 反思问题（每次回顾时问自己）

1. **我们是否偏离了初心？**
   - 当前工作重点是否符合核心使命？
   - 是否在追求"完美系统"而非"实用系统"？

2. **系统是否过于复杂？**
   - 有多少组件是真正必需的？
   - 有多少文档是真正被阅读的？
   - 能否简化而不损失功能？

3. **是否真正生效？**
   - Rules 是否每次会话都加载？
   - Skills 是否被正确识别和加载？
   - Agents 是否能被自动委派？

4. **是否在持续改进？**
   - 上周比上周更高效吗？
   - 有哪些可以自动化的手动操作？
   - 有哪些冗余可以清理？

## 🎓 学习要点

### 从本次会话学到的教训

1. **文档冗余陷阱**
   - 问题: 创建了 README.md + QUICK_START.md + SYSTEM_ARCHITECTURE.md
   - 根源: 想要"全面"而忽略了"简洁"
   - 解决: 遵循单一入口原则，README.md ≤ 800字

2. **功能目录 README 误识别**
   - 问题: agents/rules/skills 目录下放置 README.md
   - 根源: 忘记了 Lingma 会将其误识别为实例
   - 解决: 功能目录只放定义文件，文档统一放 docs/

3. **过度验证工具**
   - 问题: 创建了多个验证脚本
   - 根源: 想要"全面覆盖"
   - 解决: 合并为单一验证工具，聚焦核心检查
